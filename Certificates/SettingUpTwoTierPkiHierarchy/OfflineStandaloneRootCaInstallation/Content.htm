In this step we are going to install an offline Standalone Root CA. If a root CA is in some way compromised (broken into, hacked, stolen, or accessed by an unauthorized or malicious person), then all of the certificates that were issued by that CA are also compromised. Since certificates are used for data protection, identification, and authorization, the compromise of a CA could compromise the security of an entire organizational network. For that reason, many organizations that run internal PKIs install their root CA offline. That is, the CA is never connected to the company network, which makes the root CA an offline root CA. Make sure that you keep all CAs in secure areas with limited access. To ensure the reliability of your CA infrastructure, specify that any root and non-issuing intermediate CAs must be offline. A non-issuing CA is one that is not expected to provide certificates to client computers, network devices, and so on. This minimizes the risk of the CA private keys becoming compromised, which would in turn compromise all the certificates that were issued by the CA. To learn how to install a Standalone Root CA, proceed with the following steps...
  • First, log into the server where you are going to install the Standalone Root CA.
  • Then make certain that this server is not connected to any network. This server needs to be completely offline.
  • Once you have confirmed that this server is NOT connected to the network, begin by selecting 'Start' at the bottom left corner of your screen.
  • Then go to 'Administrative Tools'...
  • Then 'Server Manager'...
  • Once in Server Manager, select 'Roles' at the upper left corner of your screen.
  • Then, at the far right side of your screen, select 'Add Roles' to launch the Add Roles Wizard.
  • The first screen of the wizard asks you to verify if this server is properly configured.

  • Click 'Next'.
  • Select the 'Active Directory Certificate Services' option.

  • Then press 'Next >'.
  • This screen of the wizard gives a brief introduction to Active Directory Certificate Services.

  • Click 'Next >'.
  • On this screen, the wizard will ask which components of 'Active Directory Certificate Services' you want to install. In this case, we only want to install the first component, 'Certification Authority'. This is the core component, and is required in order to use 'Active Directory Certificate Services'.

  • Click 'Next >'.
  • On this next screen, you'll need to decide if the CA will be an 'Enterprise' or 'Standalone' CA. As you can see, the 'Enterprise option is greyed out. This is because this option requires the server to be a member of a domain. Therefore, since this server is not a member of a domain, this option is not available. The 'Standalone' option is what we want anyway, so we will leave it selected and move on.

  • Click 'Next >'.
  • On this screen, we need to decide if we are going to add an additional subordinate CA or install the root CA. In this case, we will leave it on the default option of 'Root CA' and move on.

  • Click 'Next >'.
  • The next screen asks if a new private key is to be created or if an existing private key is to be used. In this case, this is a new root CA, and thus a new private key needs to be created. If there was a situation where the CA had to be recovered, and had a copy of the old private key, the second option could be used to recover the root CA.

  • Click 'Next >'.
  • On the next screen, you can choose what type of encryption you want to use for the private key. This first option, 'Select a cryptographic service provider', allows you to select the algorithm that is used to create the certificate.
  • The default choice is, 'RSA#Microsoft Software Key Storage Provider'. Any provider listed that has a hash in the name, is referred to as a 'Cryptography Next Generation' or (CNG). This means more features than the non (CNG) providers, but also means less operating system support. (CNG) requires Windows Vista or Windows Server 2008 and above. This includes other CAs, for example a subordinate CA. If you are using earlier operating systems for subordinate CAs or clients, you will need to select a different option. In this case, the default will work well for this network.
  • The next option is the length of key. A larger key length will give you better security, but will increase the time taken to generate and check signatures on certificates. Also remember, that if you are planning to use the certificates with hardware like VPN devices, these devices may have limits on how long the key length can be.
  • '2048' has good software and hardware support, so we will leave it on this option and move on.
  • The next option determines the hash algorithm that is used to sign certificates.
  • The default is 'SHA1'. This once again, comes down to what your client's certificates support. The option you choose here will determine which clients can use these certificates.
  • This last option, 'Allow administrator interaction when the private key is accessed by the CA.', is an option you can select to improve the security of the private key. When you select this option, any certificate interaction performed by this server will require an administrator password to be entered in order for the operation to be performed. For example, if the CA where to issue a certificate, the administrator would be required to enter in a password before the certificate could be issued. This increases the security of the CA. On a root CA, this is not a bad option to have on, as the root CA will not be issuing too many certificates. On a CA that issues a lot of certificates to clients, you would not select this option, as it would require an administrator to enter the password every time a certificate is issued. This would quickly become a lot of work on a busy CA.
  • Click 'Next >'.
  • On the next screen, you need to enter in a common name. The wizard will automatically enter in a name for you, but you can change it to your likings if you wish.
  • In this case, we will name the CA 'BronsTutorials-RootCA'. This name will be used by all certificates that are issued from this hierarchy, and can not be changed later on. For this reason, you should choose a good name.

  • Click 'Next >'.
  • On the next screen, you need to enter in how long the certificate is valid for. The default is for '5 years'.
  • Since this is a root CA that will be secured and taken offline once it has issued the required certificates, we will enter in a value of '20 Years'. As you move down the hierarchy, you can reduce this value, but it is recommended to choose a high value for the root CA.

  • Click 'Next >'.
  • On the next screen, you need to enter in the location for the certificate database. For performance reasons, it is recommended to put these files on a separate hard disk. Since this server will only issue a few certificates, you can except the default option and store the files on the 'C' drive. On a server that issues a lot of certificates, you may want to consider storing the certificates on a separate hard disk.

  • Click 'Next >'.
  • On the next and final screen of the wizard, the wizard will display all the options you chose, and also notice there is a warning message. This message indicates that once the install is complete, you will not be able to change the computer name or domain options for this computer. For this computer, it essentially means we can not change the name of the computer and we can not add it to a domain later on if we wanted to. For this reason, ensure that you have selected an appropriate name for the server before you finish the wizard.

  • Press 'Install'.
  • Wait while the Add Roles Wizard installs 'Active Directory Certificate Services'.
  • View the installation results and then click the 'Close' button.
