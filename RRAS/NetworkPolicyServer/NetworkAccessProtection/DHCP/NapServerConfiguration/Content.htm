To learn how to configure the Network Policy Server for NAP using DHCP, proceed with the following steps...
  • Begin first by selecting 'Start' at the bottom left corner of your screen.
  • Then go to 'Administrative Tools'...
  • Then 'Network Policy Server'...
  • Once in Network Policy Server, make sure that the option selected in the dropdown is 'Network Access Protection (NAP)'.
  • Then select the link 'Configure NAP'. This will launch the 'Configure NAP' wizard.
  • Once the 'Configure NAP' wizard has started, the first thing we need to do is select the dropdown.
  • And then select the option 'Dynamic Host Configuration Protocol (DHCP)'.
  • Selecting this option will generate a name that will be used for each policy created with this wizard. You can keep this name or you can change it if you wish. In this case, we will leave the name as is and move on.
  • Press 'Next '.
  • On this next page of the wizard, we need to specify the DHCP server that will be used as the RADIUS client. To do this, press the 'Add...' button.
  • Once the 'Add' button is pressed, the first thing we need to do is enter in a name for our RADIUS client, which in this case the RADIUS client is our DHCP server.
  • In this case, we will call our RADIUS client 'DHCP Server'.
  • Once the name has been established, we next have to enter in either the IP address of the RADIUS client or its actual name. In this case, we will enter in the name which is 'dhcpserver'.
  • We can then verify it by pressing the 'Verify...' button.
  • And then pressing the 'Resolve' button.
  • If the address is able to be resolved, then it will be displayed here.
  • Press 'OK'.
  • Next, we have the option of selecting an existing Shared Secrets template, but in this case we don't have one configured, so we will move on.
  • Since we don't have a Shared Secrets template configured, we have the option of manually typing a shared secret or automatically having a shared secret generated.
  • If we select the 'Generate' option.
  • And then press the 'Generate' button.
  • Once the 'Generate' button is pressed, Windows will generate a rather long shared secret for you to use. This will give you greater security, but you will then need a way to copy this shared secret from this server to the client. Having this longer key means you now need some way of managing the key, which may mean storing the key on devices such as a USB stick or even emailing the key to transfer it between the two servers. Having a large key like this does impose more difficulty on the management of keys, but gives you much greater security.
  • A shorter key is often a lot easier to manage and can even be memorized. If we clear the generated key.
  • And select the option 'Manual'.
  • We can then manually type a shared secret here.
  • The shared secret that we type here, must be the same shared secret we enter later on the client.
  • You'll need to type the shared secret twice.
  • After the shared secret is entered, press 'OK'.
  • Now that we have our DHCP server assigned as the RADIUS client, we can move on.

  • Press 'Next'.
  • On this screen, we have the option of specifying which scopes on the DHCP server will be used with NAP. If no scopes are specified, then NAP will apply to all scopes on the DHCP server. In this case, we will not specify any scopes and move on.

  • Press 'Next'.
  • Here we can specify which computers will be effected by the NAP policy. similar to the previous screen, if no computers are specified, then the NAP policy will apply to all computers. In this case, we will not specify any computers and move on.

  • Press 'Next'.
  • On this screen of the wizard, we have the option of specifying a remediation server group. Remediation server groups are used to specify servers that are available to noncompliant NAP clients for the purpose of remediating their health state to comply with health requirements. If we already had a remediation server group configured, then we could select that group here.
  • In this case, we do not have a remediation server group configured, therefore, we will configure one now. To do this, press the 'New Group...' button.
  • After the 'New Group' button has been pressed, the first thing we need to do is give a name to our remediation group.
  • In this case, we will name our remediation group, 'Remediation Servers'.
  • Next, we need to specify which servers on the network will be part of this remediation server group. To do this, press the 'Add' button.
  • The first server that we will add will be our WSUS server. To begin, first we need to provide a friendly name for the server.
  • In this case, the name will be 'wsus'.
  • Then enter in the actual name of the server, which in this case is 'wsus'.
  • We can then verify the server by pressing the 'Resolve' button.
  • Once the 'Resolve' button is pressed, the IP address of the server will be placed here.
  • Press 'OK' to add the server to the group.
  • Another server that we want to add to the remediation server group is the Read Only Domain Controller server. To do this, press the 'Add' button again.
  • And then type in the friendly name for this server, which in this case will be 'rodc'.
  • Then type the actual name of the server, which is 'rodc'.
  • verify the name.
  • And then press 'OK'.
  • Once you have the group defined, press 'OK' to create the group.
  • If you have a web page that provides users with instructions on how to bring computers and devices into compliance with your NAP health policy, you can enter the URL here. If you do not have a web page, you can leave this field blank.
  • Press 'Next'.
  • Here we have the default System Health Validator that will be used with this health policy. If there were more System Health Validators installed, then you could choose which ones you wanted to use for this health policy here.
  • If this option, 'Enable auto-remediation of client computers' is enabled, NAP-capable client computers that are denied full access to the network because they are not compliant with the health policy, can obtain software updates from remediation servers. If this option is not enabled, noncompliant NAP-capable client computers are not automatically updated and cannnot gain full network access until they are manually updated.
  • For the old computers on your network that are not even capable of supporting NAP, for example a Windows 2000 computer, you can configure how network access will be handled for those computers in this section here.
  • The first option is to deny these computers full access to the network and only allow access to the remediation servers. This option is the default option.
  • Or this option allows these computers full access to the network.
  • Press 'Next'.
  • And then finally, on this screen, press 'Finish' to create the policy.
