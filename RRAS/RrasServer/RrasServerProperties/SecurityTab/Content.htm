To learn about the Security tab, proceed with the following steps...
  • This first section provides a lists of available authentication providers for remote access or demand-dial authentication.
  • If we select the dropdown, we can view the list of available authentication providers.
  • With this authentication provider, 'Windows Authentication', the server uses either a local account database, or a domain account database to authenticate remote access or demand-dial connection credentials. Domain account databases can be either an Active Directory database or a Windows NT 4.0 domain account database. The server logs connection authentication information in log files that are configured in the properties of the Remote Access Logging folder.
  • Or with this authentication provider, 'RADIUS Authentication', the remote access server uses a Remote Authentication Dial-In User Service (RADIUS) server to authenticate remote access or demand-dial connection credentials.
  • If the 'Radius Authentication' option is selected, we can then configure what RADIUS servers will be used for authentication. To do this, click on the 'Configure...' button.
  • This is where the RADIUS servers that will be used will be added.

  • Press 'OK'.
  • Next, we can configure the authentication methods supported by this server for remote access and demand-dial connections by clicking on the 'Authentication Methods...' button.
  • This first authentication protocol is 'Extensible authentication protocol (EAP)', and it is one of the most secured authentication protocols used today that takes advantage of TLS (Transport Layer Security). Can be implemented on a Windwos Server that is part of an Active Directory Infrastructure. EAP-TLS can be used by Routing and Remote Access clients authenticating using either smart-cards or certificates. This technology offers a secure authentication protocol because each client must store a valid X.509 certificate. This means that besides the client’s password, an attacker must get access to the private key of the certificate in order to infiltrate the network. If your company decides to store certificates within smart-cards, you increase network security even further because smart-cards can be compromised only by physical theft. Within the corporate network you would need to deploy a Certificate Authority to issue digitally signed certificates for Routing and Remote Access clients.
  • This authentication protocol, 'Microsoft encrypted authentication version 2 (MS-CHAP v2)' is enabled by default on Windows Server Editions. MS-CHAP v2 offers a two-way authentication mechanism which verifies the identity of the Routing and Remote Access server and the Routing and Remote Access client. Both parties authenticate with each other by using two different  cryptographic keys. MS-CHAP v2 uses a unique session identifier and user credentials to generate encryption keys unlike EAP-TLS which takes advantage of digital certificates.
  • Next, 'Encrypted authentication (CHAP)' is an older authentication protocol that uses MD5 hash algorithms to send the authentication requests. The RRAS Server sends a challenge to the Routing and Remote Access client which in turn responds with an MD5 hash result that is composed of the challenge and the user’s password. The RRAS Server calculates the user’s hash locally and compares the result with the hash received from the Routing and Remote Access client. If these two hashes match, then the user is granted network access. You should use this authentication mechanism if EAP-TLS is not supported within your network.
  • This next authentication protocol 'Unencrypted password (PAP)', is a simple authentication protocol that is not widely used since it uses basic authentication mechanisms and is susceptible to external attacks. PAP uses plain-text passwords that are sent to the server for authentication. You should not use this protocol for Routing and Remote Access authentication.
  • And then the last authentication protocol is 'IKEv2'. Its primary advantage is that it tolerates interruptions in the underlying network connection. For example, if the connection is temporarily lost, or if a user moves a client computer from one network to another, IKEv2 automatically restores the Routing and Remote Access after the network connection is reestablished — all without intervention on the part of the user. This feature is referred to as Routing and Remote Access Reconnect or Agile Routing and Remote Access.
