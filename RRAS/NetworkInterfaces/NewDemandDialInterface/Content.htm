Demand-dial interfaces connect routers on either public or private networks. A demand-dial interface can be either on-demand (activated only when needed) or persistent (always connected) and initiate one-way or two-way connections. Demand-dial routing is not the same as remote access. Remote access connects a single user to a network. Demand-dial routing connects networks together. However, both remote access and demand-dial routing use PPP to negotiate and authenticate the connection and encapsulate data sent on the connection. Because the routing service and remote access coexist on a server running Routing and Remote Access, both routers and remote access clients can call the same phone number. The server running Routing and Remote Access that answers the call must be able to distinguish a remote access client from a router that is calling to create a demand-dial connection. To differentiate a remote access client from a demand-dial router, the user name in the authentication credentials sent by the calling router must exactly match the name of a demand-dial interface on the answering router. Otherwise, the incoming connection is assumed to be a remote access connection. In this tutorial we'll be implementing a demand-dial interface between two remote sites, one site in NY and the other site in LA. To learn how to do this, proceed with the following steps...
  • Begin first by selecting 'Start' at the bottom left corner of your screen.
  • Then go to 'Administrative Tools'...
  • Then 'Routing and Remote Access'...
  • Once in Routing and Remote Acces, expand the server node, which in this case is 'RRASSERVER (local)'.
  • Then right-click 'Network Interfaces'.
  • And select the option 'New Demand-dial Interface...'.
  • Once you select the option 'New Demand-dial Interface..', a demand-dial interface wizard will appear. To begin the wizard, press 'Next >'.
  • On this screen of the wizard, you have the option to enter in your own name for this demand-dial interface or keep the default name.
  • For this example, we will enter in a name for this interface. A common practice is to name interfaces after the network or router to which they connect. In this case, our demand-dial interface will be connecting to the LA site, therefore we will name this demand-dial interface 'LA Site'.

  • Press 'Next >'.
  • On this next screen, you have to decide which type of demand-dial interface you want create. In this case, we do NOT want to select the option, 'Connect using PPP over Ethernet (PPPoE)', because PPPoE is used to link to the local ISP. Instead, leave the default option selected 'Connect using virtual private networking (Routing and Remote Access)'.

  • Press 'Next >'.
  • because we chose on the previous screen the option to connect using a virtual private network, we now have to choose on this screen which type of Routing and Remote Access connection we want to create. In this case, we will select the option 'Layer 2 Tunneling Protocol (L2TP)'.
  • Press 'Next >'.
  • On this screen of the wizard, you are ask to enter in the name or IP address of the router you are connecting to.
  • The router that we want this interface to connect to is the public router of the LA site. Therefore, in this case we will enter in the LA site's public IP address of '134.201.250.151'.

  • Press 'Next >'.
  • Next, on this screen, keep the option 'Route IP packets on this interface' selected. This specifies whether IP routing, the forwarding of IP packets from one routing interface to another, is enabled on this interface. You must enable IP routing in order for LAN and demand-dial routing to occur.
  • But also, we need to select the option 'Add a user account so a remote router can dial in'. This allows you to create a new user account that is used by the calling router. When this option is selected, a user account with the same name as the demand-dial interfaces is created in the security accounts database.
  • Press 'Next >'.
  • On this next screen, in order for this demand-dial connection to be activated, a static route to the remote network must be created. To do this, begin by pressing the 'Add' button.
  • After pressing the 'Add' button, a dialogue box will appear. From here is where we will create the static route.
  • First, enter in the IP address of the network this static route will be routing to. In this case, the IP address of the remote network is '192.168.100.0'.
  • Then enter in the network mask.
  • And then finally, enter in the metric. The metric is an integer value ranging from 1 to 9,999 that represents the cost of the route. This cost is used in calculating the fastest, most reliable, and/or least expensive route. For this tutorial we will only be adding one static route to this interface, so it doesn't matter what value we enter here. Therefore, in this case we will just enter in a value of '1'.

  • Press 'OK'.
  • Now that our static route has been created, we can move on...

  • Press 'Next >'.
  • On this next screen, we need to provide the password that the LA site will use when it dials in to this site (NY Site).
  • First, enter in the password. This password is the same password that is configured on the remote router. In other words, the remote router's dial-out password needs to match this router's dial-in password. Essentially, what you type here is what will be used to confirm whether or not the router dialing in is a demand-dial router.
  • Then confirm the password.

  • Press 'Next >'.
  • Now on this screen, we need to provide a user name and password for when this site (NY Site) dials in to the remote site (LA Site).
  • In this case, we will give this interface the user name 'NY Site'.
  • The domain name goes here. Typically, both sites belong to the same domain, so this field is optional.
  • And then, type in a password. Now, without trying to make this more confusing then this may already be, this password does NOT need to be the same as the password you just entered in for this interface's dial-in password. You can make this password what ever you wish, as long as this password matches the dial-in password of the remote router. In fact, all the information that you provide for the dial-out credentials on this screen must match all the dial-in credentials configured on the remote router.
  • And confirm the password.

  • Press 'Next >'.
  • And then finally, press 'Finish' to complete the 'Demand-Dial Interface Wizard'.
  • Once you complete the 'Demand-Dial Interface Wizard', a new network interface will be created.
  • Since we will be using L2TP/IPsec for this demand-dial Routing and Remote Access, we will need to configure a preshared key for our Routing and Remote Access authentication. To do this, right-click the demand-dial interface that we just created.
  • And select 'Properties'.
  • Once in the properties of our demand-dial interface, select the 'Security' tab.
  • Then select 'Advanced Settings'.
  • From here, we need to select the option 'Use preshared key for authentication'.
  • Then type in the preshared key that corresponds to the preshared key on the Routing and Remote Access server that you will be connecting to. In this case, this interface will be connecting to the Routing and Remote Access server at the LA site, which has a preshared key configured as 'LAPresharedKey'.

  • Once the preshared key has been entered, press 'OK'.

 
  • The demand-dial interface configuration is now complete, well at least it is for this site anyway. Everything that was just configured here will also have to be configured on the remote site as well. Once both sites have been configured, then a demand-dial connection can be made.

  • Press 'OK'.
