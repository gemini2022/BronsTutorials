The General tab allows you to temporarily suspend name resolution and to configure zone type (including Active Directory integration), zone file name, dynamic updates, and aging.
  • In this section, the status indicator and the associated 'Pause' button lets you see and control whether this zone can be used answer queries. When the zone status is set to 'Running', the server can use this zone to answer client queries. When the zone status is set to 'Paused', the server won't answer any queries it gets for that particular zone.

  • You can use this field to change or make a zone a primary, secondary, or stub zone. You can also use it to integrate a zone with Active Directory Domain Services. To do this, first press the 'Change...' button.
  • The first option changes the zone to a primary zone.
  • This next option changes the zone to a secondary zone.
  • And this option changes the zone to a stub zone.
  • And finally, if this DNS server is a domain controller, it changes the zone to be Active Directory-Integrated. This option is not available if the zone is changed to a secondary zone.

  • You can use this field to change the scope of replication throughout Acive Directory. To do this, first press the 'Change...' button.
  • The first option will replicate the zone data to all DNS servers running on domain controllers in the forest.
  • The next option, which is the default option, will replicate the zone data to all DNS servers running on domain controllers in the domain.
  • And this option will replicate the zone data to all domain controllers in this domain. Meaning, even if you do not install DNS on all DCs, ALL DCs in that domain will participate in replication of the zone.
  • And finally, this last option, which will replicate the zone data to all domain controllers specified in the scope of this directory partition. This option is grayed out if there are no custom DNS application partitions.

  • When using custom application partition for DNS Zones, you have 100% control of exactly which domain controllers participate in replication of the DNS information, and it makes it easier to smoothly retire a DNS server.

  • You have a choice of having dynamic updates turned off, a mixture of nonsecure and secure updates, or if the DNS server is a domain controller, having 'Secure Only' updates.
  • With this field, resource records can be aged and scavenged. To configure the option of resource records being scavenged, first press the 'Aging...' button.
  • Selecting this option will allow stale resource records to be scavenged for this zone.
  • DNS clients by default update their DNS resource records every 24 hours. There are two ways to do this. The first way is if the client’s IP has changed since the last update. If it has, this is considered an “update”. The second way is if the IP has NOT changed then it is considered a “refresh”. In both cases a new timestamp is set.

  • The No-Refresh interval is the period of time when the DNS server will NOT accept a “refresh” from the client. The DNS server will accept an "update' (where the IP changes), but NOT a "refresh'. The reason for this is to cut down on DNS replication. If you have multiple DC’s, then this makes sense. There is no point of updating a record with a new timestamp and causing unnecessary replication traffic if nothing else has changed. By setting the No-refresh interval you prevent these refreshes from taking place causing less replication.
  • The Refresh interval starts AFTER the no-refresh interval. Within this time, both an "update' and a "refresh" are allowed to happen. If a resource record hasn’t been updated by the time the no-refresh and refresh intervals have elapsed, then that resource record becomes eligible for scavenging.
