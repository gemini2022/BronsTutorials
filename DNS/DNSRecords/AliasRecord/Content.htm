A CNAME record allows the administrator to create a DNS record that effectively redirects the request to a different record. For example, if you had a CNAME record called ftp, you could configure the CName record to point to any other host record on the network. The advantage of this is it can change the CName at any time to point to a different server. The only down side of this approach is that if the client has the old record in their cache, the client will keep using this DNS record until its cache expires. To learn how to create a CNAME record, proceed with the following steps...
  • Begin first by selecting 'Start' at the bottom left corner of your screen.
  • Then go to 'Administrative Tools'...
  • Then 'DNS'...
  • Once in 'DNS', select 'View' on the top menu of the 'DNS Manager' window.
  • Then select 'Advanced'. This will unhide any additional options that are normally hidden.
  • Now, expand your server, in this case it is 'DC1'.
  • Then expand 'Forward Lookup Zones'.
  • Once 'Forward Lookup Zones' has been expanded, select a zone. In this case, we're going to select the 'bronstutorials.local' zone.
  • Now that the zone has been selected, right-click it.
  • And select 'New Alias (CNAME)...'.
  • For this example, we will create an Alias record for the server that is hosting the FTP service, so we will name the Alias record 'ftp'.
  • Next, we need to browse to the A record of the server that is hosting the FTP service. To do this, we will first press 'Browse...'.
  • We will then double-click on our DNS server 'DC1'.
  • Then double-click 'Forward Lookup Zones'.
  • Then double-click the zone of where the A record of the server that is hosting the FTP service is located. In this case it's 'bronstutorials.local'.
  • And then finally, we'll select the A record of the server that is hosting the FTP service and then press 'OK'.
  • This option 'Delete this record when it becomes stale' is an option that was made present when the 'Advanced' option in the 'View' menu was switched on. If this option is selected, the DNS server, when running its scavenging process, will remove this record (if the record is old enough, which by default is 14 days). If this option is NOT selected, this record will not be able to be removed by the scavenging process, even if options for scavenging has been configured. By default, dynamically created DNS records will automatically have this option selected, while manually created DNS records will NOT.

  • Underneath this option is this record's time stamp. This is what the scavenging process will look at to determine if the DNS record should be removed.
  • When this option is selected, it permits the resource record to be updated dynamically. When the update is performed, the host requesting the update is granted permission to modify the resource record, but all other nonadministrative permissions are removed from the access control list (ACL) that protects the resource record. This makes it possible for the administrator to create a secure resource record for a host that is not yet online and still enable the resource record to be updated dynamically when the host obtains its IP address through Dynamic Host Configuration Protocol (DHCP).

 
  • Here at the bottom is the time to live value. This value is what was also made present when the 'Advanced' option in the 'View' menu was switched on. For this DNS record, the time to live is set to one hour. This means that a DNS server will only be able to keep this DNS record in its cache for one hour before it is forced to contact an athoritative DNS server to obtain the DNS record again. This value can be edited, but you should consider how often the record will be changed. Because of caching, changes to a DNS record will not reach the entire network until the original TTL has expired - a good reason for setting a short TTL. However, caching helps reduce network traffic. The longer the TTL, the longer the record will live in other DNS server caches around the world, and so fewer requests to the original DNS server are needed - a good reason for setting a long TTL. Generally, for a record pointing to a server/device with a static IP address and no need for quick updates, a TTL of one day is a good starting point. However, if the record is for a host with a dynamic IP address or for server which is part of a failover set, you should be using a TTL value of a few minutes or less.
  • Press 'OK'.
