Use the Maintenance tab to configure mailbox database settings, including specifying a journal recipient, setting a maintenance schedule, and mounting the database at startup. To learn how to do this, proceed with the following steps...
  • Select this check box to enable transport journaling of e-mail.
  • Once selected, we next need to specify a journal recipient. To do this, click 'Browse' to open the 'Select Recipient' dialog box.
  • Once the 'Select Recipient' dialog box is open, chose the journal recipient from the following list.
  • In this case we will choose the 'Administrator' for our journal recipient, which is already selected by default.
  • Once a selection has been made, click 'OK'.
  • Use this dropdown to select one of the preset maintenance schedules.
  • Once the dropdown is selected, you can choose from the following options:
    • Run daily from 11:00 P.M. to 3:00 A.M.
    • Run daily from midnight to 4:00 A.M.
    • Run daily from 1:00 A.M. to 5:00 A.M.
    • Run daily from 2:00 A.M. to 6:00 A.M.
    • Use Custom Schedule
  • Or you can configure a custom time, to do this, click 'Customize'.
  • Then use the 'Select Schedule' dialog box to configure a customized time.
  • Once a customized time is configured, click 'OK'.
  • Select this check box to enable online database scanning, which runs continuously in the background. Online database scanning performs a checksum calculation of the database and performs operations that allow Exchange to scan for lost space on the database and recover it. If you select this check box, Exchange scans the database no more than one time per day and will issue a warning event if it can't finish scanning the database in a seven day period. 
  • Select this check box to prevent Exchange from mounting this mailbox database when it starts.
  • Select this check box to allow the mailbox database to be overwritten during a restore process.
  • Click this check box to enable circular logging. Circular logging overwrites and reuses a single log file after the data it contains has been written to the database. Circular logging is disabled by default. By enabling circular logging, you reduce drive storage space requirements. However, you can't recover anything more recent than the last full (normal) backup because the transaction logs no longer contain all the transactions that were completed since the last backup. Therefore, in a normal production environment, circular logging isn't recommended.
